## Overview

**Prague** (`Prague`) — a high-performance, compile-time-safe in-memory event cache for .NET 9, designed for event-sourced data streams from Kafka.

Key features:
- Source-generator-driven: zero-reflection, compile-time-safe fluent query API.
- Three index types: `KeyValue` (1:1), `KeyValueList` (1:N), `Range` (sorted) — automatic intersection and plan selection.
- Compile-time joins (1:1 and 1:N) with pooled, zero-allocation result sets.
- Kafka integration with change detection and conditional updates.
- Span-based, stack-allocated, SIMD-friendly hot paths (~9.7M reads/sec, <100ns indexed lookup).

## Architecture

```
src/
  Prague.Attributes/   (sic — directory name on disk) Public attributes ([DataCache], [DataCacheKey], [DataCacheIndex], [DataCacheForeignKey], ...)
  Prague.Codegen/      Roslyn source generator — emits cache classes, query builders, joins
  Prague.Core/         Runtime: InMemoryDataCache, indexing, query execution, pooled join results
                                   (T4 templates: JoinQueryBuilders.tt, JoinResults.tt → *.generated.cs)
  Prague/              Top-level package facade
  Prague.Kafka/        Kafka consumer/producer integration, SerDe, filters, background worker
  Prague.Kafka.TestAdaptor/ In-memory Kafka adaptor for tests
  Prague.Api/          HTTP API surface for inspecting caches
  Prague.Api.UI/       UI assets for the API

tests/        NUnit projects per layer (Core, DI, Generated, Kafka.Integration, Kafka.TestAdaptor, Models).
              Generated.Tests carries its own [DataCache] fixtures under Fixtures/ (Entities, Enums).
benchmarks/   Prague.Benchmarks (BenchmarkDotNet)
```

Data flow: user-defined `[DataCache]` POCO → Codegen emits `XxxCache` partial + index storage + fluent `Query()` API → at runtime `InMemoryDataCache` resolves the optimal index plan, intersects, and returns either allocating or pooled (`ExecutePooled()`) results that the caller `Dispose()`s.

Solution filters split build scope: `*.Publish.slnf` (NuGet-publishable) vs `*.Tests.slnf` (test-only).

## Detected Patterns

> Naming note: this section's text uses the current names `JoinOne` / `InnerJoinOne` / `JoinOneResolver*` / `CacheQueryBuilder_JoinOne_Extensions`. Older git history, branch names (`feature/join-one-new`), and design specs (`docs/superpowers/specs/2026-05-19-or-query-clause-design.md`, plans under `docs/superpowers/plans/`) refer to the same APIs as `JoinOneNew` / `InnerJoinOneNew` / `JoinOneNewResolver*` — the trailing `New` was dropped when the legacy `JoinOne` was retired and the new shape took its name. `JoinMany` follows the same story (legacy retired, no `New` suffix in the current code).

- **Source generation everywhere**: cache classes, query builders, and join result types are emitted by `Prague.Codegen` from `[DataCache]` attributes. Don't hand-edit `*.generated.cs`.
- **T4 templates** (`*.tt`) generate combinatorial join builders/results in `Core` — regenerate via the template rather than editing the `.generated.cs` output.
- **Pooled APIs**: query execution comes in two flavors — allocating (`Execute()`) and pooled (`ExecutePooled()` returning a disposable result). Pooled is the default for hot paths.
- **Hot-path style**: `Span<T>`, `stackalloc`, `ArrayPool`, `[InlineArray]`, `ref struct`, `Unsafe.As`, `BinaryPrimitives` are used in indexing/IO/SerDe code; preserve zero-allocation invariants when modifying.
- **Partial classes**: every `[DataCache]` user type is `partial` so the generator can extend it.
- **Kafka cache lifecycle**: `KafkaCachesBackgroundWorker` drives consumption; caches expose `ICacheAfterHandler`, `IEnrichable`, `IKafkaConfigurable` extension points.
- **Message filter pattern**: `KafkaCacheHandlerBuilder` has `WithHeaderFilter`, `WithKeyFilter(Func<TKey, bool>, bool treatAsDelete = false)`, and `WithValueFilter(Func<TValue, bool>, bool treatAsDelete = false)` builder methods. All compose with AND across multiple calls. Predicate exceptions are caught at the channel-loop call site, logged via `LoggerMessage`, and treated as reject. No-filter path is zero-alloc (`IsEmpty` inline check). Internal filter types live under `src/Prague.Kafka/Filters/`. **Key + value filters share `FilterDecision { Accept, Skip, Delete }`** (`Filters/FilterDecision.cs`): the aggregate `KafkaKeyFilters<TKey>.Evaluate(key)` / `KafkaValueFilters<TValue>.Evaluate(value)` return it; each concrete filter carries an `internal abstract bool TreatAsDelete`. First-reject-wins: the first rejecting filter's `TreatAsDelete` flag picks `Delete` vs `Skip`. `Skip` → silent drop during load / `ExecuteAfterHandlersFilter(UpdateType.Filtered)` live (unchanged legacy behaviour). `Delete` → `HandleDelete` live (removes key, fires `UpdateType.Delete` with old value only if key was present) / `_cache.Remove(key, ...)` during load (no after-handler). A thrown predicate maps to `Skip`, never `Delete`. **Header filters do NOT support `treatAsDelete`** — they are evaluated upstream in the `HeadersFiltering...` deserializer before the key is deserialized, still returning `bool`. Caveat: a key is immutable, so key-filter `treatAsDelete` only evicts a cached key when the predicate closes over mutable external state and a new message for that key arrives; for pure key predicates it is inert. Consumer eval sites: key filter at `KafkaCacheConsumer.cs` ~314 (both phases), value filter in `ChannelLoop` ~349 (live) and `FlushBuffer` ~249 (load).
- **Health checks (`Prague.Kafka.Health`)**: split liveness/readiness `IHealthCheck` implementations backed by a pure `KafkaCachesHealthEvaluator` (no DI, testable in isolation). State lives on existing `KafkaCachesConsumerStatistics` / `KafkaDataCacheStatistics` as plain fields (no `Volatile`/`Interlocked`). Healthy path is zero-allocation — allocate only on failure. Register via `AddPragueKafkaLiveness` / `AddPragueKafkaReadiness` on `IHealthChecksBuilder`.
- **`*Unsafe` accessor convention**: raw, unsynchronized reads on `KafkaCachesConsumerStatistics` are exposed via `*Unsafe` properties (`BrokerUpCountUnsafe`, `LastPollTimestampUnsafe`, `IsFatalLatchedUnsafe`, `HasLostPartitionsUnsafe`). The backing fields are `internal` — only the `*Unsafe` accessors are public API. Follow this pattern when adding new observable-but-unlocked state.
- **MessagePack-first numeric header SerDe**: `HeadersSerDe.SerializeMessagePack<T>` is the canonical write path for `int`/`int?`/`long`/`long?` Kafka headers (Dericher, codegen). On the read path, always use `TryDeserializeMessagePackExact<T>` first — **not** `TryDeserializeMessagePack<T>` — because the non-exact overload silently misparses raw `BitConverter` bytes (it reads the first byte as a positive fixint without consuming all bytes, returning a wrong value without throwing). `TryDeserializeMessagePackExact<T>` uses the `ReadOnlyMemory<byte>` overload with `out int bytesRead` and only returns `true` when `bytesRead == bytes.Length`. After the exact check fails, fall back to `TryDeserializeInt` (4-byte raw) / `TryDeserializeLong` (8-byte raw) for backward compat. This ordering is implemented in all 5 filter classes (`KafkaHeaderEqualsFilter<T>`, `KafkaHeaderNotEqualsFilter<T>`, `KafkaHeaderEqualsNumericFilter`, `KafkaHeaderNotEqualsNumericFilter`, `KafkaHeaderPredicateFilter<T>`) and the codegen Enricher. `SerializeInt`/`SerializeLong`/`TryDeserializeInt`/`TryDeserializeLong` are public API and must not be deleted.
- **TestAdaptor in-memory Kafka cluster**: `AddKafkaCacheTestCluster()` on `IServiceCollection` wires up `KafkaCacheTestBuilderProvider` (implements both `IKafkaCacheBuilderProvider` and `IKafkaCacheTestBuilderProvider`) as a drop-in for real Kafka. Key behaviours: `InjectDump(path)` loads a MessagePack `.pkd` file and replays its records followed by a partition EOF; `InjectDumps(path)` batch-loads a directory of `.pkd` files; `AddCacheTopics()` auto-registers topics from `IDataCacheRegistry` (deferred to `Init(IServiceProvider)` via `_initActions`). `TestKafkaCluster` distributes messages round-robin among consumers sharing a `group.id`; consumers without a configured `group.id` get a unique UUID (independent consumer semantics). `Consume(int millisecondsTimeout)` returns `null` on timeout — does not throw — matching real Confluent Kafka consumer behaviour. `KafkaCacheTestBuilderProviderMarshall` is a partial static class that bridges `internal` interface members for callers in other assemblies; its `AddCacheTopicsImpl` partial method is hooked by source-generated code.
- **`PragueMessagePack` isolation pattern**: All internal MessagePack call sites pass `PragueMessagePack.Options` explicitly — never `MessagePackSerializer.DefaultOptions`. Host mutations of `DefaultOptions` must not affect Prague's wire format (verified by `IsolationFromDefaultOptionsTests` which poisons `DefaultOptions` with a `ThrowingResolver` and asserts `HeadersSerDe`, `CacheSerde`, and `KafkaHeaderPredicateFilter<DateTime>` still work). `PragueMessagePack` is a static class in `Prague.Kafka` with a sentinel pattern: `_defaultSentinel` is initialized via `CreateDefaultComposite()`; `_options` is initialized to it. `CreateDefaultComposite()` (internal static) builds a `CompositeResolver` with the built-in `MessagePack.Formatters.StringInterningFormatter` in the formatter slot (uses `ConditionalWeakTable` internally — interned strings are GC-reclaimable, no intern-table leak) alongside `PragueDateTimeResolver` and `TypelessContractlessStandardResolver`. `Configure(options)` overwrites only when still at sentinel, is idempotent on same reference, and throws `InvalidOperationException("conflicting")` on second call with a different reference. **`Configure` is called inline at DI-registration time** (inside `AddKafkaCaches`) — not deferred to `BuildServiceProvider` — so `PragueMessagePack.Options` is active immediately after `services.AddKafkaCaches(...)` returns. `ResetForTests()` (internal) restores the sentinel; tests must call it in both `[SetUp]` and `[TearDown]`. `PragueDateTimeFormatter` (sealed) dispatches reads by `reader.NextMessagePackType`: `Integer` → `DateTime.FromBinary(ReadInt64())`, `Extension` → `reader.ReadDateTime()`, otherwise throws. Host composition via `KafkaCachesGlobalOptionsBuilder.WithMessagePackResolver(Func<IFormatterResolver, IFormatterResolver>)` — `Build()` delegates to `PragueMessagePack.CreateDefaultComposite()` so the lambda always receives all three layers (StringInterningFormatter + PragueDateTimeResolver + TypelessContractlessStandardResolver). Compliance check: `grep -rn "MessagePackSerializer\.(Serialize|Deserialize)" --include="*.cs" src/ | grep -v "PragueMessagePack.Options"` must return zero results.
- **`IDataCache<TCache, TKey, TValue>` interface**: exposes both `Cache { get; }` (`InMemoryDataCache<TKey, TValue>`) and `Query()`. Resolver code accesses the inner cache directly via `rightCache.Cache` — do not go through `rightCache.Query()._leftQuery._dataCache`. Codegen emits `Cache` as a **property** (`public InMemoryDataCache<TKey, TValue> Cache { get; }`) not a readonly field; this satisfies the interface contract on all generated cache classes.
- **`JoinOne` (PK-to-PK outer join — wrapper-direct API)**: Level-0 join: left PK == right PK (`TLeftKey == TRightKey` enforced by the extension signature). Three overloads in `CacheQueryBuilder.JoinOne.Extensions.cs` (`CacheQueryBuilder_JoinOne_Extensions`): (1) `.JoinOne(_bCache)` — no filter; (2) `.JoinOne(_bCache, q => q.WithStatus("active"))` — optional filter; (3) `.JoinOne(_bCache, arg, static (q, a) => q.WithStatus(a))` — `TArg` user-state parameter for zero-allocation static-lambda filters. Pass the cache wrapper directly — no `.Query()` at call site. Extension parameter is `IDataCache<TRightCache, TKey, TRightValue>` (the interface, not `TRightCache`) — C# can't infer `TRightValue` upward through constraints, so the interface form lets type inference work; cast to `(TRightCache)rightCache` inside. **Single unified resolver** `JoinOneResolver<TLeftKey, TLeftValue, TRightCache, TRightKey, TRightValue, TFilter, TSelector>` (7 params) parameterized by both a `TFilter : struct, IJoinOneFilter<NonExecBuilder>` strategy struct and a `TSelector : struct, IKeySelector<TLeftKey, TRightKey>` key-selector strategy struct. All existing PK-to-PK overloads pass `IdentitySelector<TKey>` as the default selector — zero cost, `TLeftKey == TRightKey` constraint preserved at call site. **`IKeySelector<TIn, TOut>` pattern**: static-abstract `IsIdentity` property is JIT-folded per closed generic — identity path uses `Unsafe.As<TLeftKey, TRightKey>` reinterpret (zero-cost), selector path calls `Selector.Select(lk)`. Three selector structs: `IdentitySelector<T>` (identity, JIT-elides Select entirely), `KeySelector<TIn, TOut>` (wraps `Func<TIn, TOut>`), `KeySelectorWithArg<TIn, TArg, TOut>` (wraps func + arg; declare `static` for zero-alloc capture). **PK-to-PK + key selector family (6 overloads)**: `.JoinOne(keySelector, [selectorArg,] rightCache, [filter, [filterArg]])` — `keySelector: Func<TKey, TRightKey>` maps left PK to right PK. Two selector structs: `KeySelector<TIn, TOut>` (wraps `Func<TIn, TOut>`) and `KeySelectorWithArg<TIn, TArg, TOut>` (wraps func + arg; use `static` for zero-alloc capture). Enables cross-key-type joins (e.g. `int → long`) without new resolver types. Tested in `JoinOneKeySelectorTests.cs`. Three filter strategy structs: `NoFilter<TBuilder>` (identity, JIT-elides Apply entirely), `JoinOneFilter<TBuilder>` (wraps `Func<TBuilder, TBuilder>`), `JoinOneFilterWithArg<TBuilder, TArg>` (wraps `Func<TBuilder, TArg, TBuilder>` + arg field; declare lambda `static` for zero-alloc capture). `IJoinOneFilter<TBuilder>` interface has single `Apply(TBuilder q)` method; all three structs are `readonly` with `[AggressiveInlining]` on Apply — JIT devirtualizes per closed generic, `NoFilter.Apply` is fully elided. `AsNonExecutable()` (internal extension on `CacheQueryBuilderCombined` in `CacheQueryBuilderCombinedDiscriminatorExtensions`) swaps the discriminator `ExecutableQuery<TCache>` → `NonExecutableQuery<TCache>` preserving all other generic parameters — call `Cache.Query().AsNonExecutable()` in any resolver needing a non-executable builder. Filter callback receives `NonExecutableQuery<TRightCache>` discriminator — codegen `WithXxx` extensions callable, `Execute*` hidden. Execution: builds `ValueSet<JoinedKeyPair<TLeftKey, TRightKey>>` via identity (`Unsafe.As`) or selector per-key, wraps in `PairedCacheQueryBuilderCoreCombined`, applies filter, calls `ExecutePaired(ref container)`. Pairs are guarded by the `handedOff` leak-safety pattern — see paired-builder Detected Pattern for full detail. **`InnerJoinOne` — unified post-walk design (all 4 resolver families, landed)**: `InnerJoinOne(rightCache[, filter[, arg]])` at level 0 constructs the resolver with `isInner: true`; 3 overloads exist for PK-to-PK, RightUnique, and LeftUnique (LeftSym deferred). `PrepareIndexedInner` calls `_ = leftQuery.GetCandidates<TLeftKey>()` to trigger auto-populate-from-leftCache. **`UnsafeExecuteIndexedInner` flow (all 3 implemented resolver families)**: (1) seed `ValueSet<JoinedKeyPair>` from candidates; (2) `Filter.Apply` (configures `_filter` predicate, narrows `_candidates` via `UseIndex` if present); (3) `ExecutePaired(ref container)` — walks pairs, `container.Add` for matches only; (4) `accessor.RetainNonNullSlots<TLeftKey, TRightValue>(ref candidates)` — drops `_results` entries whose Right_N slot is null/default (miss or predicate-reject), narrows `candidates` to surviving keys. A slot is non-null IFF this resolver wrote to it via `container.Add`; the `is null` check handles ref types and `Nullable<T>` uniformly. LeftSym chained-inner deferred (T4 emission covers only minimal Shape A identity Inner no-filter; full surface pending). Open scope: Phase C (LeftSym chained-inner full surface), selector-inner overloads at level 0, Phase 1C (index/list-index chained variants).
- **`JoinOne` symmetric-index variant (left-index-driven outer join)**: Triggered by `[DataCacheIndex(DataCacheIndexType.Many, Symmetric = true)]` — codegen emits `CacheSymmetricKeyValueListIndex` instead of `CacheKeyValueListIndex`. `[DataCacheIndex(DataCacheIndexType.Unique, Symmetric = true)]` on the left entity emits `CacheSymmetricUniqueIndex` (subclass of `CacheUniqueIndex` with `.Reverse` property) — handled by the separate left-unique-index variant below. The symmetric index maintains a `.Reverse` dictionary (leftKey → indexValue) enabling O(1) reverse lookup. Base overloads (identity selector): 2 shapes × 3 filter variants = 6 overloads. Selector overloads (Shape A + key selector, Shape B + key selector): additional 6 + 6 overloads. **Shape A** (no rightIndex): `.JoinOne(leftSymIndex, rightCache, [filter])` — identity: `TLookupKey == TRightKey`; with selector: `.JoinOne(leftSymIndex, keySelector, rightCache, [filter])` — selector maps `TLookupKey → TRightKey`. **Shape B** (with rightIndex): `.JoinOne(leftSymIndex, rightCache, rightIndex, [filter])` — identity; with selector: `.JoinOne(leftSymIndex, keySelector, rightCache, rightIndex, [filter])` — selector maps `TLookupKey → TRightIndexKey`. Resolver `JoinOneLeftSymResolver<TLeftKey, TLeftValue, TRightCache, TLookupKey, TRightIndexKey, TRightKey, TRightValue, TFilter, TSelector>` (9 params) carries both `TFilter : struct, IJoinOneFilter<...>` and `TSelector : struct, IKeySelector<TLookupKey, TRightIndexKey>`. Identity overloads pass `IdentitySelector<TLookupKey>` — zero cost. **Borrow-the-set execution (supersedes Phase β-3 `FanOutContainer<TInner>` / side-map design)**: `TLeft` for the paired core is `LeftKeySetView<TLeftKey>` (public wrapper around the internal `PooledSet<TLeftKey>`) — pairs are `JoinedKeyPair<LeftKeySetView<TLeftKey>, TRightKey>` where `JoinedKey` wraps a **borrowed reference** to the symmetric index's internal `PooledSet<TLeftKey>` for that lookupKey (no copy, no side map). `CacheSymmetricKeyValueListIndex` exposes 8 bulk primitives (`IntersectValues` / `IntersectValuesVia`, each in `ref ValueSet` + `ReadOnlySpan` × identity + selector variants): each iterates candidates, calls `_cacheReverse.TryGetValue(lk, out lku)` → `GetValuesUnsafe(lku)` → emits `new JoinedKeyPair<LeftKeySetView<TKey>, TRightKey>(new LeftKeySetView<TKey>(borrowedSet), rk)`; `ValueSet` dedups on `.Key` so multiple lefts sharing a lookupKey collapse to one pair. Dispatch helpers `SeedPairsFromSpan` / `SeedPairsFromSet` route to the correct primitive by `RightIndex is null` (Shape A vs B) and `TSelector.IsIdentity`. Two fan-out containers replace the old single `FanOutContainer<TInner>`: `OuterFanOutContainer<TAccessor>` reinterprets the view back to `PooledSet<TLeftKey>` via `Unsafe.As` then iterates it, writes via `GetValueRef` + skip-if-null (outer-attach contract — slots pre-exist); `InnerFanOutContainer<TAccessor>` does the same reinterpret, then filters via `_candidates.Contains(lk)` (correctness invariant: borrowed set may include lefts not in the query candidate set), creates slots via `GetValueRefOrAddDefault`. The old `rightToRepr: ValueDictionary<TRightKey, TLeftKey>` and `groupsByRepr: ValueDictionary<TLeftKey, ValueSet<TLeftKey>>` are gone — zero extra allocations. Safe `IJoinResolver` variants (`ExecuteWithAccessor`, `ExecuteIndexedInner`) throw `NotSupportedException` — chain only calls Unsafe paths. `TFilter` constraint type-arg: `PairedCacheQueryBuilderCoreCombined<LeftKeySetView<TLeftKey>, TRightKey, TRightValue>`; all 24 LeftSym extension overloads use `LeftKeySetView<TLeftKey>` as the `TLeft` generic. Multiple left entities sharing the same index value all receive the same right value (or `null` on miss/filter-out). Core-layer counterpart (no codegen): `JoinOneSymIndexCoreTests.cs` (`Prague.Core.Tests/Join/`) — NUnit tests covering Shape A and Shape B including fan-out and filter variants.
- **`JoinOne` right-unique-index variant (FK-on-right outer join)**: Triggered when the right entity carries a unique FK back to the left. Codegen emits `CacheUniqueIndex<TRightKey, TRightValue, TLeftKey>` (subclass of `CacheKeyValueIndex<>`) from `[DataCacheIndex(DataCacheIndexType.Unique)] TLeftKey Yyy` on the right entity. Three identity overloads: (1) `.JoinOne(rightCache, rightUniqueIndex)` — no filter; (2) with filter; (3) filter + TArg. Three selector overloads: `.JoinOne(keySelector, [selectorArg,] rightCache, rightUniqueIndex, [filter, [filterArg]])` — selector maps `TLeftKey → TIndexKey` where `rightUniqueIndex` is keyed by `TIndexKey`. Resolver `JoinOneRightUniqueIndexResolver<TLeftKey, TLeftValue, TRightCache, TRightKey, TIndexKey, TRightValue, TFilter, TSelector>` (8 params); identity overloads pass `IdentitySelector<TLeftKey>`. **Execution (Phase β-2)**: `RightIndex.IntersectValues(leftKeys, ref pairs, add: true)` builds `ValueSet<JoinedKeyPair<TLeftKey, TRightKey>>` in one bulk call (use the non-generic overload at `InMemoryDataCache.cs:205` — the generic overload at line 225 sets `.JoinedKey = default!` and is wrong here) → wrap in `PairedCacheQueryBuilderCoreCombined` + `CacheQueryBuilderCombined<NonExecutableQuery<TRightCache>, PairedCoreType, ...>` → `Filter.Apply(builder)` (executor-agnostic, discriminator-gated) → `builder._leftQuery.ExecutePaired(ref container)`. No `FanOutContainer` needed (1:1 unique-index guarantee, no fan-out). Canonical use case: `Book → BookInfo` where `BookInfo.BookId` is a unique index. 5 NUnit tests in `JoinOneRightUniqueIndexTests.cs` (Generated.Tests); Core-layer counterpart (no codegen) in `JoinOneRightUniqueIndexCoreTests.cs` (`Prague.Core.Tests/Join/`).
- **`JoinOne` left-unique-index variant (FK-on-left, 1:1 symmetric unique index)**: Triggered by `[DataCacheIndex(DataCacheIndexType.Unique, Symmetric = true)]` on the LEFT entity. Codegen emits `CacheSymmetricUniqueIndex<TLeftKey, TLeftValue, TIndexKey>` (distinct from `CacheSymmetricKeyValueListIndex` used by `Many` symmetric indices). Its `.Reverse` property is itself an index supporting `IntersectValues`. Three identity overloads (Shapes L1-L3): `.JoinOne(leftSymUniqueIndex, rightCache, [filter, [filterArg]])`. Three selector overloads: `.JoinOne(leftSymUniqueIndex, keySelector, [selectorArg,] rightCache, [filter, [filterArg]])` — selector maps the index value (`TIndexKey` from `.Reverse`) → `TRightKey`. Resolver `JoinOneLeftUniqueIndexResolver<TLeftKey, TLeftValue, TRightCache, TIndexKey, TRightKey, TRightValue, TFilter, TSelector>` (8 params) where `TSelector : struct, IKeySelector<TIndexKey, TRightKey>`; identity overloads pass `IdentitySelector<TIndexKey>`. **Execution**: identity path: `LeftIndex.Reverse.IntersectValues(leftKeys, ref pairsAsIndex, add: true)` via `Unsafe.As` reinterpret (`TIndexKey == TRightKey` at call site); selector path: `LeftIndex.Reverse.IntersectValuesChain<TRightKey, TSelector>(leftKeys, Selector, ref pairs, add: true)` — two-step chain `leftKey → Reverse.TryGetValue → TIndexKey → Selector.Select → TRightKey`. Builds `ValueSet<JoinedKeyPair<TLeftKey, TRightKey>>` then `PairedCacheQueryBuilderCoreCombined` → `Filter.Apply` → `ExecutePaired(ref container)` (bijective 1:1 — no `FanOutContainer` required). Canonical use case: `Author.BookId` (unique symmetric index, long) with selector `long → int` to reach `Book` (int PK). NUnit tests in `JoinOneLeftUniqueIndexTests.cs`; selector-variant tests in `JoinOneKeySelectorTests.cs` (`Prague.Generated.Tests.Join`). Core-layer counterparts: `JoinOneLeftUniqueIndexCoreTests.cs` and `JoinOneKeySelectorCoreTests.cs` (`Prague.Core.Tests/Join/`).
- **`CacheKeyValueIndex` bulk-intersect struct-selector overloads**: Three additional overloads on `CacheKeyValueIndex<TKey, TValue, TIndexKey>` support JIT-devirtualized key transformation without delegate allocation. `IntersectValues<TOtherValue, TSelector>(ReadOnlySpan<TOtherValue>, TSelector, ref ValueSet<JoinedKeyPair<TOtherValue, TKey>>, bool)` — struct-dispatched selector transforms each source value to `TIndexKey`; original source value (not the transformed key) is stored as `JoinedKey`, preserving caller identity through the join; `IdentitySelector<T>` overloads fold to the no-selector identity path at JIT time. `IntersectValues<TOtherValue, TSelector>(ref ValueSet<TOtherValue>, TSelector, ref ValueSet<JoinedKeyPair<TOtherValue, TKey>>, bool)` — `ref ValueSet` companion to the above `ReadOnlySpan` overload; same struct-dispatched selector semantics, used by `InnerJoinOne` selector paths (avoids converting the candidates `ValueSet` to a span, keeping the loop body inside the index where the JIT can devirtualize per closed generic). `IntersectValuesChain<TOutKey, TSelector>(ReadOnlySpan<TIndexKey>, TSelector, ref ValueSet<JoinedKeyPair<TIndexKey, TOutKey>>, bool)` — tail selector applied AFTER `TryGetValue`; two-step chain `(input) → TryGetValue → TKey → tailSelector.Select → TOutKey`; used by `JoinOneLeftUniqueIndexResolver` for `Reverse[leftKey] → idx → Selector.Select(idx) → rightKey`. All three are `[AggressiveInlining]`; `TSelector` must be `struct, IKeySelector<TIn, TOut>`.
- **`LeftKeySetView<T>` public wrapper / `PooledSet<T>` internal**: `PooledSet<T>` (`Collections/PooledSet.cs`) is `internal sealed class`. The paired-core `TLeft` slot for the LeftSym resolver uses `LeftKeySetView<T>` (`src/Prague.Core/LeftKeySetView.cs`) — a `public readonly struct` with a single `internal readonly PooledSet<T> Set` field and an internal-only constructor. Zero public members; opaque to callers. It exists solely so `JoinOneLeftSymResolver`'s filter constraint and `PairedCacheQueryBuilderCoreCombined<LeftKeySetView<TLeftKey>, ...>` can appear in public generic signatures without violating CS0703.
